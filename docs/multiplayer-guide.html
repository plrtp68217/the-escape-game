Godot 4 · C# · High-Level Multiplayer API

Мультиплеер в Godot: что, как и зачем

Полное пособие по встроенному сетевому API Godot — от транспорта и RPC до автоматической репликации узлов. Каждый инструмент разобран по схеме «что это → как применять → зачем именно так», а в конце всё собрано на примере реального кода игры The Escape Game.

01

Модели сети и главная идея

Прежде чем трогать API, нужно решить, кто кому доверяет. От этого зависит буквально каждая строчка сетевого кода.

Peer-to-peer против клиент-сервера

В Godot любая машина в сессии — это peer (участник) со своим числовым id. Есть два подхода к тому, кто главный:

  • Полный P2P — все равны, каждый рассылает своё состояние всем. Просто, но открыто читам: любой клиент может «сказать», что убил всех.
  • Клиент-сервер с авторитетом сервера — один peer (сервер, id = 1) принимает решения, остальные лишь просят и отображают. Так делают почти все сетевые игры.
Сервер (id 1)
Единственный источник правды. Валидирует действия, хранит и рассылает состояние мира.
⇄peers
Клиенты
Шлют ввод/запросы, получают решения, интерполируют и рисуют.
Зачем

Авторитет — это про доверие, а не про производительность. Если решение может изменить исход игры (урон, подбор, счёт), его должен принимать сервер. Клиент только просит. Это одно правило снимает 90% будущих проблем рассинхрона и читов.

Хост — это сервер + клиент одновременно

В кооперативных играх часто нет выделенного сервера. Тогда один из игроков — хост: у него запущен и сервер (IsServer() == true), и его собственный игровой персонаж. Именно эта модель используется в проекте.

02

Транспорт: MultiplayerPeer

Нижний слой — как байты физически летают между машинами. Godot прячет это за интерфейсом MultiplayerPeer, и вы выбираете реализацию.

РеализацияТранспортКогда брать
ENetMultiplayerPeerUDP (ENet)Десктоп/мобайл. Быстро, есть надёжные и ненадёжные каналы. Дефолт.
WebSocketMultiplayerPeerTCP/WebSocketЭкспорт в браузер (HTML5), где UDP недоступен.
WebRTCMultiplayerPeerUDP (WebRTC)Браузер + P2P без своего сервера (нужен сигналинг).
OfflineMultiplayerPeer—Одиночный режим: тот же код работает «в сети из одного».

Создание сервера и клиента

// ХОСТ: поднять сервер
var peer = new ENetMultiplayerPeer();
Error err = peer.CreateServer(port: 7777, maxClients: 8);
if (err != Error.Ok) return err;              // порт занят и т.п.
Multiplayer.MultiplayerPeer = peer;           // ← вот здесь сеть «оживает»

// КЛИЕНТ: подключиться
var peer = new ENetMultiplayerPeer();
peer.CreateClient("127.0.0.1", 7777);
Multiplayer.MultiplayerPeer = peer;           // соединение пойдёт асинхронно
Момент

Ключевая строка — присваивание Multiplayer.MultiplayerPeer = peer. До неё это просто сокет. После — включается High-Level API: начинают приходить сигналы, работают RPC, спавнеры и синхронизаторы.

Важно

CreateClient возвращается сразу, ещё до установки соединения. Успех/провал придёт позже сигналом ConnectedToServer / ConnectionFailed. Не спавните ничего сразу после вызова.

03

MultiplayerAPI: центральный объект

Свойство Node.Multiplayer у любого узла возвращает MultiplayerAPI — «пульт» сети. Через него — сигналы, id, роли и вызовы RPC.

ЧленЧто возвращает / делает
GetUniqueId()Мой peer id. У сервера всегда 1, у клиентов — случайные большие числа.
IsServer()true, если этот процесс — сервер (или хост). Главный «гейт» для решений.
GetRemoteSenderId()Внутри RPC-метода: id того, кто прислал вызов. Вне RPC — 0.
GetPeers()Список id всех подключённых peer'ов (без себя).
MultiplayerPeerТот самый транспорт. Присвоить = включить; null = отключиться.
Зачем

Один и тот же код запускается на всех машинах. IsServer() и GetUniqueId() — это способ внутри общего кода понять «а я сейчас кто?» и разветвить поведение: сервер считает, клиент просит.

04

Сигналы жизненного цикла

Сеть событийна. Вы подписываетесь на сигналы Multiplayer в _Ready() и реагируете на подключения и обрывы. Подписаться нужно один раз (обычно в autoload-синглтоне).

СигналГде срабатываетТипичная реакция
PeerConnected(id)У всех, чаще важен серверуСервер спавнит игрока для нового id, шлёт ему состояние.
PeerDisconnected(id)У всехСервер удаляет узел игрока — репликация снимет его у всех.
ConnectedToServer()Только у клиентаСоединение удалось → переход в лобби/игру.
ConnectionFailed()Только у клиентаНе дозвонились → сообщение и возврат в меню.
ServerDisconnected()Только у клиентаХост упал/закрылся → выкинуть в меню с причиной.
 
// NetworkManager._Ready() — подписка один раз при старте
Multiplayer.PeerConnected      += OnPeerConnected;
Multiplayer.PeerDisconnected   += OnPeerDisconnected;
Multiplayer.ConnectedToServer  += OnConnectedToServer;
Multiplayer.ConnectionFailed   += OnConnectionFailed;
Multiplayer.ServerDisconnected += OnServerDisconnected;

private void OnPeerConnected(long id) {
    if (!Multiplayer.IsServer()) return;   // спавнит только сервер
    SpawnPlayer((int)id);
}
Грабли

PeerConnected приходит и хосту, и клиентам. Почти всегда обработчик должен начинаться с if (!IsServer()) return; — иначе клиенты попытаются спавнить чужих игроков и всё задублируется.

05

RPC — вызвать метод на другой машине

RPC (Remote Procedure Call) — способ вручную синхронизировать события: «игрок выстрелил», «стена сломалась», «имя изменилось». Помечаете метод атрибутом — и вызываете его удалённо.

Атрибут и режимы доступа

[Rpc(MultiplayerApi.RpcMode.AnyPeer)]          // кто ВПРАВЕ вызвать удалённо
private void SetReady(bool ready) { … }

[Rpc(MultiplayerApi.RpcMode.Authority, CallLocal = true,
     TransferMode = MultiplayerPeer.TransferModeEnum.Reliable)]
private void BroadcastState(int hp) { … }
ПараметрЗначениеСмысл
RpcMode.Authorityпо умолчаниюВызвать вправе только владелец узла (обычно сервер). «Сервер оглашает».
RpcMode.AnyPeer—Вызвать может любой. «Клиент просит сервер». Всегда проверяйте отправителя!
CallLocalfalse / trueВыполнить метод ещё и у себя. Нужно, чтобы у хоста тоже сыграл эффект.
TransferMode.Reliable—Гарантированная доставка и порядок. Для важных событий (сломалось, умер).
TransferMode.Unreliable—Может потеряться. Для частого потока, где важен только свежий пакет.
TransferMode.UnreliableOrdered—Может теряться, но старые не приходят после новых.

Адресация: Rpc против RpcId

  • Rpc(nameof(Method), args…) — вызвать у всех подключённых.
  • RpcId(1, nameof(Method), args…) — вызвать только у сервера (peer 1). Так клиент шлёт запрос.
  • RpcId(id, …) — вызвать у конкретного peer (например, отдать состояние лобби только что вошедшему).

Канонический паттерн «просьба → решение → оглашение»

  1. 1
    ClientAnyPeer → 1
    RpcId(1, nameof(RequestFire))

    Клиент не стреляет «сам» — просит сервер. Локально можно проиграть предсказание (см. гл. 09).

  2. 2
    Server
    if (!IsServer()) return; var who = GetRemoteSenderId();

    Сервер проверяет отправителя и правила (жив ли, есть ли патроны), меняет своё состояние — источник правды.

  3. 3
    ServerAuthority → все
    Rpc(nameof(SpawnBullet), who, dir)

    Сервер оглашает результат всем (с CallLocal, чтобы и у хоста появилась пуля).

Безопасность

Любой AnyPeer-метод — это дверь снаружи. Первая строка — if (!IsServer()) return;, вторая — валидация через GetRemoteSenderId(). Никогда не доверяйте параметрам клиента вслепую.

06

Авторитет узлов

У каждого узла есть «владелец» — peer, который вправе им управлять и чьё состояние считается истинным. Это и есть механизм, разделяющий «мой персонаж» и «чужой».

ВызовЧто делает
SetMultiplayerAuthority(id, recursive=true)Назначить владельца узла. По умолчанию рекурсивно — охватывает детей (в т.ч. Synchronizer).
GetMultiplayerAuthority()Узнать id владельца.
IsMultiplayerAuthority()true, если я владелец. Гейт для ввода, физики, камеры.
 
// Player.cs — управляет собой только владелец
public override void _PhysicsProcess(double delta) {
    if (!IsMultiplayerAuthority()) return;   // чужих не двигаем — их везёт Synchronizer
    // … читаем WASD, гравитация, MoveAndSlide()
}

private void ApplyAuthority() {
    SetMultiplayerAuthority(PlayerId);        // рекурсивно → и дочерний Synchronizer
    _camera.Current = IsMultiplayerAuthority(); // своя камера только у себя
}
Тайминг

Нельзя звать SetMultiplayerAuthority прямо в _Ready() спавненного узла — движок ещё не закончил репликацию, будет ошибка «no network ID». Откладывайте на кадр через CallDeferred(nameof(ApplyAuthority)). Метод вызывается на каждом peer, чтобы все согласованно знали владельца.

Зачем

Авторитет отвязывает «кто вычисляет» от «кто видит». Владелец считает движение и пишет позицию; MultiplayerSynchronizer рассылает её остальным, которые физику не трогают. Без этого два игрока дрались бы за одну и ту же переменную позиции.

07

MultiplayerSpawner: появление и удаление

Узел, который автоматически реплицирует создание и удаление дочерних узлов в заданной ветке. Добавили ребёнка на сервере — он появился у всех. Никаких ручных RPC на спавн.

Настройка

  • Spawn Path — ветка, за которой Spawner следит (например, ../Players).
  • Spawnable Scenes — белый список сцен, которые разрешено спавнить (безопасность: клиент не заставит сервер спавнить что попало).
  • Spawn Limit — необязательный лимит числа узлов.
Сервер
Players.AddChild(player, true)
→Spawner
авто
MultiplayerSpawner
Реплицирует узел всем текущим и будущим peer'ам
→
Все клиенты
Узел появляется в их дереве сам
// Спавнит ТОЛЬКО сервер. Второй аргумент true = уникальное имя.
private void SpawnPlayer(int id) {
    var player = GD.Load<PackedScene>("res://scenes/player.tscn").Instantiate<Player>();
    player.Name = id.ToString();
    player.PlayerId = id;
    GetNode("/root/Main/Players").AddChild(player, true);   // → Spawner подхватит
}

Авто- и кастомный спавн

По умолчанию Spawner реплицирует узел «как есть» с экспортируемыми свойствами. Если нужна логика при создании (позиция из пула точек, случайный цвет), задают SpawnFunction и вызывают spawner.Spawn(data) — data (любой Variant) прилетит в функцию на каждом peer.

Зачем

Spawner решает «проблему позднего входа»: подключился клиент в середине матча — и все уже существующие игроки/предметы автоматически появляются у него. Руками рассылать «а ещё есть вот эти» не нужно.

08

MultiplayerSynchronizer: непрерывное состояние

Если Spawner отвечает за «есть/нет», то Synchronizer — за «какой прямо сейчас». Он раз за разом рассылает выбранные свойства узла от владельца всем остальным.

Как настраивается

У Synchronizer есть ресурс SceneReplicationConfig — список свойств и режим для каждого. Для каждого свойства задаются два независимых момента:

  • spawn — послать значение один раз при появлении узла (стартовая позиция, id).
  • replication mode — как синхронизировать дальше.
РежимПоведениеПример
Never (0)Не синхронизировать после спавнаPlayerId — задан раз и навсегда
Always (1)Слать каждый сетевой тикposition, rotation движущегося игрока
OnChange (2)Слать только при измененииздоровье, состояние (открыт/закрыт)

Так настроен игрок в проекте (player.tscn): position и rotation — Always, PlayerId — только spawn. Дополнительно у Synchronizer есть частота (ReplicationInterval) и дельта-интервал, а также фильтры видимости (SetVisibilityFor) — можно не слать узел тем, кто его не видит (туман войны, оптимизация).

RPC или Sync?

Synchronizer — для того, что «всегда имеет значение прямо сейчас» и переживает поздний вход (позиция, HP). RPC — для мгновенных событий, которые случились в момент («выстрел», «дверь хлопнула»). Часто их комбинируют.

Направление

Synchronizer шлёт от владельца узла к остальным. Поэтому важно, чтобы authority узла (и его, рекурсивно) был у правильного peer. Если позиция «дёргается» — почти всегда перепутан авторитет.

09

Практики плавной синхронизации

Сеть не мгновенна: пакеты идут десятки миллисекунд и иногда теряются. Чтобы игра не «телепортировала» — несколько приёмов поверх базового API.

Интерполяция чужих объектов

Позиции приходят рывками (10–30 раз в секунду), а экран рисует 60+. Чужие объекты не ставят в присланную точку мгновенно, а плавно доводят к ней между пакетами. Простейший вариант — Lerp к целевой позиции в _Process.

Предсказание на клиенте (client-side prediction)

Ждать ответа сервера на каждый шаг — это заметный лаг управления. Владелец применяет ввод сразу локально и параллельно шлёт его серверу; когда приходит авторитетный ответ, позицию мягко корректируют. Так своё движение ощущается мгновенным, оставаясь под контролем сервера.

Надёжность под нагрузку

  • Поток позиций — Unreliable: потерянный пакет неважен, следующий свежее.
  • Разовые события (урон, смерть, подбор) — Reliable: их нельзя терять.
  • Не синхронизируйте то, что можно вычислить локально из уже синхронного состояния.
  • Дробите по каналам (TransferChannel), чтобы тяжёлый поток не блокировал важные события.
Принцип

Меньше данных по сети — устойчивее игра. Сначала спросите «а можно это не слать?». Синхронизируйте минимальный набор истины, остальное каждый peer выведет сам.

10

Всё вместе: как это в игре

Соберём инструменты в живой поток на примере The Escape Game. Две autoload-подсистемы + узлы репликации в сцене.

/root
├─ NetworkManager   autoload — peer, сигналы, спавн игроков
├─ LobbyManager     autoload — список игроков и готовность (RPC)
└─ Main            сцена: мир + UI
   ├─ Players       ← сервер добавляет сюда игроков
   ├─ MultiplayerSpawner  spawn_path = ../Players
   └─ UserInterface меню / лобби / табло
  1. 1
    Host
    CreateServer → MultiplayerPeer → RegisterHost → SpawnPlayer(1)

    Хост поднимает сервер, регистрирует себя в лобби и спавнит свой персонаж. Гл. 02, 07.

  2. 2
    Clienthandshake
    CreateClient → signal ConnectedToServer / PeerConnected

    Клиент подключается; сервер ловит PeerConnected(id). Гл. 03, 04.

  3. 3
    HostSpawner
    SpawnPlayer(id) → Players.AddChild → репликация всем

    Сервер спавнит игрока для нового id; Spawner раздаёт узлы всем, включая уже существующих. Гл. 07.

  4. 4
    ВсеSync + Authority
    SetMultiplayerAuthority(id) · Synchronizer гонит position/rotation

    Каждый двигает только свой персонаж; позиции разъезжаются по сети автоматически. Гл. 06, 08.

  5. 5
    Clientлобби
    RpcId(1, SetPlayerReady) → Rpc(SyncPlayerReady) → GameStarting

    Готовность и старт матча — вручную через RPC «просьба → оглашение». Гл. 05.

Итого

Spawner снимает ручную работу с «кто есть в мире», Synchronizer — с «где они», RPC остаётся для событий и лобби, а IsServer() + authority держат всю логику под контролем одной машины.

11

Грабли и отладка

Ошибки, на которые натыкаются почти все. Держите этот список под рукой при первом рассинхроне.

СимптомПричина и лечение
«no network ID» при спавнеAuthority назначен слишком рано. Перенесите в CallDeferred после _Ready.
Игроки задублировалисьСпавн сработал не только на сервере. Добавьте if (!IsServer()) return;.
Чужой игрок «дёргается»/стоитПерепутан authority узла, или Synchronizer шлёт не от того peer. Проверьте рекурсивную установку.
Событие не дошло до части игроковИспользован чистый RPC при позднем входе. Держите состояние в синхронизируемом свойстве.
Эффект есть у клиентов, но не у хостаЗабыт CallLocal = true на Authority-RPC.
Клиент управляет чужимиНет гейта IsMultiplayerAuthority() во вводе/физике.
Spawner молчитСцена не в списке Spawnable Scenes, либо узел добавлен не в Spawn Path.
Отладка

Запускайте два экземпляра игры (в Godot: Debug → Run Multiple Instances). Логируйте GetUniqueId() и IsServer() в ключевых точках — большинство багов сети это «код выполнился не на той машине».
