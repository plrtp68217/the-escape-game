The Escape Game · C# · Godot 4

Как игра синхронизируется между хостом и клиентами

Разбор по шагам: какие узлы и вызовы Godot High-Level Multiplayer API работают при создании хоста, подключении клиента, спавне и синхронизации игроков, предметов на карте и ломаемых стен — с указанием, что и в какой момент срабатывает.

00

Из чего собрана сеть

Два автозагружаемых синглтона держат состояние сети и лобби (переживают перезагрузку сцены). Внутри сцены main.tscn живут узлы репликации Godot, которые автоматически рассылают спавны и состояние.

NetworkManager
Autoload. Поднимает ENet-peer, ловит сигналы Multiplayer.*, спавнит игроков на сервере.
·
LobbyManager
Autoload. Список игроков, готовность, старт матча. Всё через RPC, авторитет — сервер.
·
MultiplayerSpawner
В main.tscn. Что сервер добавит в Players — само появится у всех.
·
MultiplayerSynchronizer
Внутри player.tscn. Гонит position/rotation каждый тик от владельца остальным.
/root
├─ NetworkManager        (autoload — сеть)
├─ LobbyManager          (autoload — лобби, RPC)
└─ Main                  (сцена: мир + UI)
   ├─ Ground / Wall …    статичная геометрия, грузится одинаково у всех
   ├─ Players            ← сюда сервер добавляет игроков
   ├─ MultiplayerSpawner    spawn_path = ../Players
   └─ UserInterface      меню / лобби / табло

player.tscn (spawnable)
   Player (CharacterBody3D)          authority = peer владельца
   └─ MultiplayerSynchronizer  position(Always) rotation(Always) PlayerId(spawn)
Модель

Client–server с авторитетом сервера. Хост одновременно и сервер (peer id = 1), и игрок. Клиенты не спавнят и не удаляют узлы напрямую — они лишь просят сервер, а решения принимает и рассылает он.

01

Создание хоста

Кнопка «Создать игру» → NetworkManager.CreateHost(). Хост становится сервером с peer id = 1 и сразу спавнит самого себя.

  1. 1
    HostGodot API
    new ENetMultiplayerPeer().CreateServer(Port, MaxPlayers)

    Создаётся ENet-сервер на порту 7777 до 8 игроков. Пока это просто открытый сокет — движок ещё не знает, что это «мультиплеер».

  2. 2
    Host
    Multiplayer.MultiplayerPeer = peer

    Ключевой момент: как только peer присвоен, High-Level API «оживает». Multiplayer.IsServer() становится true, а GetUniqueId() возвращает 1. Начинают приходить сигналы.

  3. 3
    HostLobby
    LobbyManager.RegisterHost()

    Хост заносит сам себя в список лобби (id = 1, не готов). Пока он один — рассылать некому.

  4. 4
    HostSpawn
    SpawnPlayer(GetUniqueId()) → Players.AddChild(player, true)

    Инстанцируется player.tscn, ему задаются Name = id, позиция и PlayerId, и он добавляется в контейнер Players. MultiplayerSpawner увидит нового ребёнка и позже реплицирует его каждому подключившемуся.

  5. 5
    HostEvent
    Connected?.Invoke() → Main.EnterLobby()

    Внутреннее C#-событие (не сеть) переводит UI хоста в экран лобби. Хост готов принимать подключения.

02

Подключение клиента

Кнопка «Подключиться» → JoinServer(ip). Тут появляются шаги «по сети»: они выделены пунктиром — данные реально уходят между машинами.

  1. 1
    Client
    new ENetMultiplayerPeer().CreateClient(ip, Port) → Multiplayer.MultiplayerPeer = peer

    Клиент открывает соединение. Handshake идёт асинхронно — метод возвращается сразу, результат придёт сигналом.

  2. ⇄
    По сетиENet handshake
    signal: ConnectedToServer (у клиента)  +  PeerConnected(id) (у сервера)

    Соединение установилось. Клиент получает ConnectedToServer → входит в лобби. Сервер получает PeerConnected(id) с новым peer id клиента.

  3. 2
    HostSpawn
    OnPeerConnected(id) → SpawnPlayer(id) // только если игра не началась

    Сервер спавнит узел игрока для нового клиента в Players. Если матч уже идёт — вместо спавна сервер шлёт клиенту NotifyGameInProgress, и тот возвращается в меню.

  4. ⇄
    MultiplayerSpawnerПо сети
    Spawner реплицирует всех Player-узлов новому клиенту

    Spawner автоматически рассылает: у нового клиента появляются его собственный игрок и все уже существующие. Обратно — новый игрок появляется у всех остальных. Вручную ничего слать не нужно.

  5. ⇄
    RPCLobby
    RpcId(id, RegisterPlayer) · SyncFullLobbyTo(id) · Rpc(RegisterPlayer)

    Сервер отправляет новому клиенту полный слепок лобби (имена + готовность), а остальным — сообщение «добавился игрок N». Список синхронизирован у всех.

Обрыв

Сигнал PeerDisconnected (у сервера) и ServerDisconnected (у клиентов) запускают обратный путь: сервер удаляет узел игрока — Spawner снимает его у всех; клиенты при падении хоста возвращаются в меню с причиной.

03

Синхронизация игроков

Спавн решает Spawner, движение — Synchronizer. Разделяет их понятие authority: у каждого узла игрока владелец — peer этого игрока, и только он двигает свою модель.

Владелец узла
IsMultiplayerAuthority() == true. Читает ввод, считает физику, пишет position/rotation.
→Synchronizer
каждый тик
MultiplayerSynchronizer
Забирает свойства из SceneReplicationConfig у владельца и рассылает их всем остальным peer'ам.
→применяется
Все остальные
Authority == false. Ввод/физику не считают — просто принимают позицию и рисуют чужую модель.

Что именно реплицируется, задано прямо в player.tscn в SceneReplicationConfig:

СвойствоРежимКогда передаётся
positionAlwaysкаждый сетевой тик — плавное движение у всех
rotationAlwaysкаждый тик — куда игрок смотрит
PlayerIdspawnодин раз при спавне (дальше не меняется)
 
// Player.cs — авторитет назначается уже ПОСЛЕ завершения спавна,
// иначе SetMultiplayerAuthority падает с "no network ID".
public override void _Ready() {
    CallDeferred(nameof(ApplyAuthority));   // откладываем на кадр
}
private void ApplyAuthority() {
    SetMultiplayerAuthority(PlayerId);       // рекурсивно → и на Synchronizer
    _camera.Current = IsMultiplayerAuthority();  // своя камера только у себя
}

public override void _PhysicsProcess(double delta) {
    if (!IsMultiplayerAuthority()) return;   // чужих не двигаем
    // … WASD, гравитация, MoveAndSlide()
}
Момент

SetMultiplayerAuthority вызывается на каждом peer (из _Ready спавненного узла) и рекурсивно охватывает дочерний Synchronizer — поэтому все машины согласованно знают, кто «владеет» этим игроком и от кого принимать его позицию.

04

Лобби и RPC: имена, готовность, старт

Список игроков — это не игровые узлы, а обычные данные. Их синхронизируют вручную через RPC (Remote Procedure Call) — вызов метода на другой машине. Тут важны два режима доступа и направление вызова.

RpcMode.AnyPeer
«Просьба» клиента серверу: SetPlayerReady, SetPlayerName. Клиент шлёт RpcId(1, …) — только серверу. Сервер проверяет GetRemoteSenderId() и решает.
RpcMode.Authority
«Оглашение» сервера всем: RegisterPlayer, SyncPlayerReady, GameStarting. Вызвать вправе только сервер, выполняется у всех клиентов.
  1. 1
    ClientAnyPeer
    RpcId(1, nameof(SetPlayerReady), true)

    Игрок нажал «Готов». Вызов адресован только серверу (peer 1).

  2. 2
    Host
    id = GetRemoteSenderId(); _players[id].IsReady = true

    Сервер — единственный источник правды — обновляет своё состояние. По GetRemoteSenderId() он знает, кто прислал.

  3. ⇄
    HostAuthority → все
    Rpc(nameof(SyncPlayerReady), id, true)

    Сервер оглашает изменение всем клиентам, у каждого обновляется список → LobbyUpdated перерисовывает UI.

  4. ⇄
    HostGameStarting
    StartGame() → Rpc(nameof(GameStarting))

    Хост жмёт «Начать». RPC переводит всех в фазу Gameplay одновременно. Флаг _isGameStarted закрывает вход опоздавшим.

05

Предметы, лежащие на карте

Здесь выбор зависит от того, статичный предмет или динамический. Это разные механизмы Godot — важно не путать.

Статичный предмет
Лежит в main.tscn изначально (как Wall, Ground). Синхронизация бесплатна: каждый peer грузит одну и ту же сцену → предмет уже у всех. Сеть не нужна, пока он не меняется.
Динамический предмет
Появляется/исчезает во время игры (лут, выпавший ключ). Нужен MultiplayerSpawner с этим предметом в списке spawnable — сервер добавляет узел, Spawner рассылает.

Рецепт для подбираемого предмета (по образцу игроков):

  1. 1
    SetupSpawner
    MultiplayerSpawner.AddSpawnableScene("res://…/item.tscn")

    Регистрируем сцену предмета как spawnable — на отдельном спавнере или на существующем с нужным spawn_path.

  2. 2
    HostSpawn
    itemsNode.AddChild(item, true) // только на сервере

    Предмет спавнит только сервер. Spawner реплицирует его всем. Начальную позицию везёт SceneReplicationConfig (spawn=true).

  3. ⇄
    ClientAnyPeer
    RpcId(1, nameof(RequestPickup), itemId)

    Игрок «подобрал» — но не удаляет узел сам, а просит сервер. Иначе предмет исчезнет у одного и останется у других (рассинхрон).

  4. ⇄
    HostDespawn
    item.QueueFree() // на сервере → Spawner снимает у всех

    Сервер проверяет (успел ли кто раньше, в радиусе ли игрок) и удаляет узел. Spawner автоматически убирает предмет на всех клиентах.

Правило

Спавнит и деспавнит только сервер. Клиент всегда действует через запрос (AnyPeer-RPC серверу). Состояние (кол-во патронов, открыт ли сундук) держите в свойстве под MultiplayerSynchronizer или синхронизируйте Authority-RPC.

06

Ломаемая стена — «сломалось у всех»

Сейчас wall.tscn — статичный StaticBody3D без скрипта: одинаков у всех, но сломать нельзя. Чтобы разрушение отражалось у всех, добавляем стене состояние и проводим удар через сервер. Классический путь «клиент просит → сервер решает → сервер оглашает».

Удар
Игрок-authority бьёт стену. Не меняет её локально — шлёт запрос.
→RpcId(1)
запрос
Сервер
Единственный, кто считает HP: Health -= dmg. Проверяет читы/дистанцию.
→Rpc()
всем
Все клиенты
Принимают новое HP / факт разрушения → проигрывают эффект, убирают коллизию.
  1. 1
    ClientAnyPeer → сервер
    RpcId(1, nameof(RequestHit), wallPath, damage)

    Атакующий сообщает серверу «я ударил вот эту стену». Локально стена пока не меняется — ждём решения сервера, чтобы не разошлось.

  2. 2
    Host
    Health -= damage; bool broken = Health <= 0;

    Сервер — источник правды по HP. Валидирует (жива ли стена, в радиусе ли игрок), решает, разрушена ли она. Клиенты этот расчёт не делают.

  3. 3
    HostAuthority → все
    Rpc(nameof(ApplyDamage), Health); if (broken) Rpc(nameof(Break));

    Сервер оглашает результат всем (включая себя). RPC помечен CallLocal, чтобы разрушение сыграло и у хоста.

  4. 4
    ВсеОдинаково
    CollisionShape.Disabled = true; Mesh.Visible = false; spawn debris

    Каждый peer применяет один и тот же эффект. Стена исчезает синхронно у всех — потому что команда пришла из одного места.

// Wall.cs — набросок. Стене нужен скрипт + сетевой id узла.
[Rpc(MultiplayerApi.RpcMode.AnyPeer)]          // зовёт клиент → у сервера
private void RequestHit(int damage) {
    if (!Multiplayer.IsServer()) return;
    Health -= damage;
    Rpc(nameof(ApplyDamage), Health);        // огласить всем
    if (Health <= 0) Rpc(nameof(Break));
}

[Rpc(MultiplayerApi.RpcMode.Authority, CallLocal = true)]  // сервер → все + сам
private void Break() {
    _collision.Disabled = true;
    _mesh.Visible = false;
    // … частицы обломков, звук
}
RPC или Sync?

RPC — для событий («сломалась сейчас»). MultiplayerSynchronizer — для непрерывного состояния (полоска HP, доля трещин). Для стены обычно хватает RPC-события + опционально синхронизации HP, чтобы опоздавшие увидели уже подбитую стену.

Опоздавшие

Чистый RPC получают только те, кто в сети сейчас. Если стену можно сломать до входа игрока — держите Health/IsBroken в синхронизируемом свойстве, тогда состояние приедет при спавне/входе.
